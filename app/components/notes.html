🔒 Authentication Middleware

Next.js Route Protection & Session Management

Next.js 14 Middleware Auth 
import { NextRequest, NextResponse } from "next/server";
import { getSession } from "./lib/auth/auth";

export default async function proxy(request: NextRequest) {
  // Get user session
  const session = await getSession();

  // Get current path
  const pathname = request.nextUrl.pathname;

  // Define route types
  const isAuthPage =
    pathname.startsWith("/sign-in") || pathname.startsWith("/sign-up");

  const isDashboardPage = pathname.startsWith("/dashboard");

  // Logged in users should not see auth pages
  if (isAuthPage && session?.user) {
    return NextResponse.redirect(new URL("/dashboard", request.url));
  }

  // Not logged in users should not access dashboard
  if (isDashboardPage && !session?.user) {
    return NextResponse.redirect(new URL("/sign-in", request.url));
  }

  // Allow request to proceed
  return NextResponse.next();
}

export const config = {
  matcher: ["/dashboard/:path*", "/sign-in", "/sign-up"],
};

⚙️ Matcher Configuration

"/dashboard/:path*" "/sign-in" "/sign-up"

Explanation: The middleware only runs on these paths for performance optimization.

Pattern Matches
/dashboard/:path* /dashboard, /dashboard/members, /dashboard/settings, etc.
/sign-in Exactly /sign-in
/sign-up Exactly /sign-up

✅ Key Features

  • Session-based authentication
  • Protects dashboard routes
  • Prevents logged-in users from seeing auth pages
  • Optimized with matcher config
  • Runs before page render

📦 Dependencies

  • next/server - NextRequest, NextResponse
  • ./lib/auth/auth - getSession() function

📌 How It Works

  1. Intercepts requests to matched routes
  2. Retrieves user session
  3. Checks current pathname
  4. Applies redirect rules based on auth state
  5. Either redirects or allows request

⚠️ Important Notes

  • Middleware runs on the edge
  • Keep it lightweight
  • Matcher optimizes performance
  • Session must be available
